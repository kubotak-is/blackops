# P16-005: Generated Status Capability

Status: Ready

## Goal

Phase 15のimmutable Generated Operation Objectへ、一回だけFramework標準Status Resourceを取得する`.status(operationId, options)`を追加する。

Operation固有のOutcome Metadataを使い、7 State、Typed Outcome、Retry Hint、Authentication／Unavailable／Expired／Internal／TransportをFramework-neutralなStrict TypeScript Discriminated UnionへDecodeする。`.fetch()`は202後に自動Status取得せず、`.wait()`とPollingは追加しない。

## In Scope

- 全Generated HTTP Operation Objectの`.status(operationId, options)`
- Operation固有Status／Status Result Type
- `GET /operations/{operationId}`の共通Client Runtime
- Canonical UUIDv7のLocal検証
- Per-call `baseUrl`／Headers／Credentials／Signal／Injected Fetchの再利用
- 7 State Schema Version 1のStrict Decode
- Operation Type／Operation ID一致検証
- Completed Typed Outcome／EmptyOutcome Decode
- Non-terminal `Retry-After`正整数秒のDecode
- 401 Authentication、404 Unavailable、410 Expired、500 InternalのTyped Failure
- Missing Fetch、Invalid Base URL、Network、Abort、Unexpected Response、Invalid Operation IDのTransport Result
- Raw Body／Credential／Sensitive／Exception Detail非露出
- Deterministic Generated Bytes、Ownership Marker／Atomic Replace／Drift回帰
- Temporary Generated TreeのStrict TypeScript／Node Runtime検証
- Internal Documentation、Report、TODO、STATE同期

## Out of Scope

- `.wait()`、Timer、Polling、Backoff、Timeout
- `.fetch()`からの自動Status取得
- 401／404／410／5xx／Network Errorの自動Retry
- Global Mutable Client／Credential／Cache／Status Store
- Frontend Fixture／GitHub ActionsのStatus専用Assertion追加
- Quickstart／Skeleton Frontend Source、Guide、Website、Consumer E2E
- React／Vue／Svelte／Inertia Adapter、Hook、Form Helper

## Operation Object Surface

各HTTP Operation Objectへ次を追加する。

```ts
CreateOrder.status(operationId, options?);
```

- `operationId`はStringを受け、送信前にCanonical lowercase UUIDv7として検証する
- `options`は既存`OperationCallOptions`と同じ呼出単位`baseUrl`、Headers、Credentials、Signal、Fetchを使う
- `.status()`はValueを受けず、Operation RouteのPath／Query／Header／Body Bindingを使わない
- `.status()`は一回だけGETし、Pollingしない
- `.fetch()`、`.toRequest()`、`.url()`、Readonly Metadataを変更しない
- Callable、Thenable、Class Constructor、Module Import時の通信を追加しない

## Generated Status Types

各Operation ModuleはOperation固有`<Operation>StatusResult`を生成する。FoundはLifecycle Stateを`kind`と`data.state`の両方でLiteral Narrowingでき、業務上のRejected／Failed／Dead LetteredもQuery成功として`ok: true`を返す。

概念Shapeは次である。

```ts
type CreateOrderStatusResult =
  | Readonly<{
      ok: true;
      kind: 'accepted';
      status: 200;
      data: Readonly<{
        schemaVersion: 1;
        operationId: string;
        operationType: 'order.create';
        state: 'accepted';
      }>;
      retryAfterSeconds: number;
    }>
  | Readonly<{
      ok: true;
      kind: 'running';
      status: 200;
      data: Readonly<{
        schemaVersion: 1;
        operationId: string;
        operationType: 'order.create';
        state: 'running';
        attempt: number;
      }>;
      retryAfterSeconds: number;
    }>
  | Readonly<{
      ok: true;
      kind: 'retry_scheduled';
      status: 200;
      data: Readonly<{
        schemaVersion: 1;
        operationId: string;
        operationType: 'order.create';
        state: 'retry_scheduled';
        attempt: number;
        retryAt: string;
      }>;
      retryAfterSeconds: number;
    }>
  | Readonly<{
      ok: true;
      kind: 'completed';
      status: 200;
      data: Readonly<{
        schemaVersion: 1;
        operationId: string;
        operationType: 'order.create';
        state: 'completed';
        outcome: CreateOrderOutcome;
      }>;
    }>
  | Readonly<{ ok: true; kind: 'rejected'; status: 200; data: CreateOrderRejectedStatus }>
  | Readonly<{ ok: true; kind: 'failed'; status: 200; data: CreateOrderFailedStatus }>
  | Readonly<{ ok: true; kind: 'dead_lettered'; status: 200; data: CreateOrderDeadLetteredStatus }>
  | Readonly<{ ok: false; kind: 'authentication'; status: 401; error: Readonly<{ code: string }> }>
  | Readonly<{ ok: false; kind: 'unavailable'; status: 404; error: Readonly<{ code: 'operation_unavailable' }> }>
  | Readonly<{ ok: false; kind: 'expired'; status: 410; error: Readonly<{ code: 'operation_expired' }> }>
  | Readonly<{ ok: false; kind: 'internal'; status: 500; error: Readonly<{ code: 'internal_error' }> }>
  | Readonly<{ ok: false; kind: 'transport'; status: null; error: OperationStatusTransportError }>;
```

- `attempt`と`retryAfterSeconds`は正のSafe Integer
- `retryAt`はマイクロ秒付きRFC 3339 UTC Stringとして厳密検証する
- Rejectedは`error.category`／`error.code`の非空Stringだけ
- Failedは`error.code === 'operation_failed'`だけ
- Dead Letteredは`error.code === 'operation_dead_lettered'`だけ
- Terminal Found Resultは`retryAfterSeconds`を持たない
- EmptyOutcome OperationのCompleted Wire `outcome: {}`はGenerated Resultの`outcome: undefined`へDecodeする
- OutcomeありOperationはPhase 15と同じOperation固有Scalar／Nullable Decoderを使う

型Aliasの責務分割や名前は生成可読性のため変更できるが、公開されるNarrowing ShapeとInvariantは上記と同等以上にする。

## Status Request Runtime

共通`client.ts`へStatus取得を追加し、Operation Moduleへ通信実装を複製しない。

```text
validate canonical UUIDv7
  -> build /operations/{operationId}
    -> apply per-call baseUrl／headers／credentials／signal
      -> select injected fetch or globalThis.fetch
        -> one GET
          -> snapshot headers/body
            -> strict decode
```

- URLは既存Base URL Resolverを再利用し、Optional Base Pathの後ろへ`/operations/{id}`を解決する
- Request MethodはGET、BodyとFramework管理`Content-Type`は持たない
- Call Option HeaderをCase-insensitiveに重複させない既存規則を維持する
- Operation RouteのHeader BindingをStatus Requestへ流用しない
- Invalid IDはFetchを呼ばずTransport `invalid_operation_id`
- Missing Fetch／Invalid Base URL／Network／Abort／Unexpected Responseは既存Transport境界を再利用する
- Status Transport Error Codeへ`invalid_operation_id`を追加する。既存`.fetch()`のTransport Unionを不要に広げない
- URL、Header、Credential、Operation ID、Exception、Raw BodyをTransport Errorへ含めない

## Strict HTTP Decode

### Common

- 204を除く既存Fetch Runtimeと同じ`application/json` Media Type検査を使う
- JSON Object以外、Unknown／Missing Key、Getter／`text()`異常を成功へCastしない
- 200は`schemaVersion === 1`、Response `operationId`がRequested IDと一致、`operationType`がGenerated ObjectのReadonly `type`と一致することを要求する
- ResponseのCanonical UUIDv7とOperation TypeをRequest情報で補完しない
- 200のState固有でないFieldを拒否する
- ResultとNested Data／Errorを既存方針どおりFreezeする

### Retry Header

- `accepted`、`running`、`retry_scheduled`は`Retry-After`がCanonicalな正整数秒で必須
- Terminal 200は`Retry-After`が存在すればUnexpected Response
- `0`、負数、空白、符号、小数、HTTP Date、Unsafe Integer、重複を正規化して受理しない
- `retryAfterSeconds`はWire Bodyへ混ぜずResultのHintとして返す

### State Body

| State | Exact additional body |
| --- | --- |
| `accepted` | なし |
| `running` | `attempt` |
| `retry_scheduled` | `attempt`、`retryAt` |
| `completed` | `outcome` |
| `rejected` | `error.category`、`error.code` |
| `failed` | `error.code=operation_failed` |
| `dead_lettered` | `error.code=operation_dead_lettered` |

### Error Status

- 401は既存Authentication Middleware Shape `status=error`、`category=unauthorized`、非空`code`を要求する
- 404はExact `{status: 'error', code: 'operation_unavailable'}`
- 410はExact `{status: 'error', code: 'operation_expired'}`
- 500はExact `{status: 'error', code: 'internal_error'}`
- Error ResultへOperation ID、Actor、Raw Body、Unknown Fieldを保持しない
- 403、422、その他StatusまたはShape不一致はTransport `unexpected_response`

## Marker, Determinism, and Drift

Generated Runtime Surface変更としてOwnership Marker Schemaを3へ上げる。Ownership判定だけ既知Legacy 1／2／Current 3を受理し、新規／Temporary Tree検証はCurrent 3だけを受理する。

- Frontend Contract Manifest Schemaは上げない
- 同じContractから二回生成したFile Path／Bytesを一致させる
- Current 2 Treeを3へ安全に置換できる
- Unknown Marker、Non-marker、Symlink、Rollback Safetyを維持する
- `frontend:check`は新Expected Treeに対するDriftを検出する
- Generated TreeへTimestamp、Absolute Path、Environment、Credential、Fixture Valueを含めない

## Required Runtime Evidence

Temporary Generated Treeを既存Pinned TypeScript ToolchainでDOMなしStrict Compileし、生成JavaScriptで少なくとも次を検証する。

- 7 StateとState別Narrowing／Exact Keys
- Typed Outcome Scalar／NullableとEmptyOutcome `{}`→`undefined`
- Requested Operation ID／Generated Operation Type不一致
- Schema Version、Attempt、Retry At、Retry-Afterの不正
- 401／404／410／500分類
- Invalid IDでFetch 0回
- Missing／Injected／Global Fetch、Base URL、Header、Credential、Signal
- Network／Abort／Malformed／Raw Error非露出
- `.status()`が一回だけFetchし、`.fetch()`から自動実行されない
- Module Import時に通信しない
- Object／Result／Data Freeze
- `.wait()`、Timer、Polling、Global Mutable Clientが存在しない

Permanent `tests/Frontend` Fixture／CIへのStatus専用Assertion追加はP16-006で行う。P16-005ではTemporary TreeのEvidenceをPHP Test／Reportへ記録し、生成ArtifactをCleanupする。
