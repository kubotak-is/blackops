# P15-004: Typed Fetch Runtime and Results

Status: Ready

## Goal

P15-003のFramework-neutral TypeScript ESM Treeを拡張し、各immutable Operation Objectへ`.fetch()`を追加する。

Frontend Contract Manifestだけを正本として、Browser既定FetchとSSR／Node／TestのInjected Fetchを同じ共通Client Runtimeへ接続する。HTTP ResponseはStatusとJSON Shapeを検証し、Inline Completed、Void、Deferred Accepted、Protocol、Rejected、Validation、Internal、Transportを`ok`／`kind`／`status`でNarrowingできるOperation固有Typed Resultへ変換する。

## Structural Fetch Contract

Generated Typeは`lib.dom.d.ts`、Node固有Type、Frontend Frameworkへ依存させない。少なくとも次の責務をStructural Typeで表現する。

```ts
export type OperationFetchHeaders = Readonly<{
  get(name: string): string | null;
}>;

export type OperationFetchResponse = Readonly<{
  status: number;
  headers: OperationFetchHeaders;
  text(): Promise<string>;
}>;

export type OperationFetchRequest = Readonly<{
  method: string;
  headers: Readonly<Record<string, string>>;
  body?: string;
  credentials?: OperationCredentials;
  signal?: OperationAbortSignal;
}>;

export type OperationFetch = (
  url: string,
  request: OperationFetchRequest,
) => Promise<OperationFetchResponse>;

export type OperationCallOptions = OperationRequestOptions & Readonly<{
  fetch?: OperationFetch;
}>;
```

`globalThis`は`{ fetch?: OperationFetch }`として限定的に参照し、Fetchが存在しないRuntimeでもModule Loadを失敗させない。Injected Fetchを優先し、存在しない場合だけBrowser既定Fetchを使う。GlobalへClient、Base URL、Credentialを保存しない。

`OperationRequest`からFetchへ渡すのはURLとMethod／Headers／Optional Body／Credentials／Signalだけとする。Operation Objectごとに送信処理を複製せず、共通Client Runtimeを使用する。

## Generated Result Types

共通Typeは少なくとも次を表現する。

```ts
export type DeferredAcknowledgement = Readonly<{
  operationId: string;
  acceptedAt: string;
}>;

export type ProtocolError = Readonly<{
  code: string;
}>;

export type OperationRejection = Readonly<{
  category: 'business_rule' | 'unauthorized' | 'forbidden' | 'not_found' | 'conflict';
  code: string;
  operationId?: string;
}>;

export type ValidationViolation<TField extends string> = Readonly<{
  field: TField;
  rule: string;
  code: string;
}>;

export type ValidationRejection<TField extends string> = Readonly<{
  operationId: string;
  code: 'validation.failed';
  violations: readonly ValidationViolation<TField>[];
}>;

export type InternalOperationError = Readonly<{
  code: 'internal_error';
  operationId?: string;
}>;

export type OperationTransportError = Readonly<{
  code: 'missing_fetch' | 'invalid_base_url' | 'network_error' | 'aborted' | 'unexpected_response';
}>;
```

各Operation ModuleはOperation固有Outcome Type、Value Field Name Union、Result Unionを生成する。例:

```ts
export type CreateOrderOutcome = Readonly<{
  orderId: string;
}>;

export type CreateOrderResult =
  | Readonly<{ ok: true; kind: 'completed'; status: 200; data: CreateOrderOutcome }>
  | Readonly<{ ok: false; kind: 'protocol'; status: 400; error: ProtocolError }>
  | Readonly<{ ok: false; kind: 'rejected'; status: 400 | 401 | 403 | 404 | 409; error: OperationRejection }>
  | Readonly<{ ok: false; kind: 'validation'; status: 422; error: ValidationRejection<CreateOrderField> }>
  | Readonly<{ ok: false; kind: 'internal'; status: 500; error: InternalOperationError }>
  | Readonly<{ ok: false; kind: 'transport'; status: null; error: OperationTransportError }>;
```

- Inline Outcome OperationはCompleted 200を含め、Completed 204／Accepted 202を含めない
- Inline Void Operationは`{ ok: true; kind: 'completed'; status: 204; data: undefined }`を含める
- Deferred Operationは`{ ok: true; kind: 'accepted'; status: 202; data: DeferredAcknowledgement }`を含める
- Failure側5種は全Operationへ含める
- Validation FieldはOperationValue Constructor Parameter名のString Literal Unionとする。Fieldがない場合も安全な空Unionを生成する
- Operation Objectは`.fetch(value, options?: OperationCallOptions): Promise<OperationResult>`を持つ
- `.toRequest()`は`OperationRequestOptions`のまま送信機能を持たず、`.url()`とReadonly Metadataも不変とする
- Callable、Thenable、Class Constructorを生成しない

## Response Decode Contract

ResponseはStatusだけで成功型へCastしない。204以外は`Content-Type`のMedia TypeがCase-insensitiveな`application/json`であること、Bodyが有効なJSON Objectであること、Statusに対応するDiscriminantと既知Fieldが正しいことを検証する。`application/json; charset=utf-8`は許可する。

### Success

- Inline Outcome: Status 200、JSON Object、Artifactの全Outcome FieldをRequiredとして検証する
- Inline Void: Status 204、Bodyは空でなければならない
- Deferred: Status 202、`status === 'accepted'`、`operationId`と`acceptedAt`がString
- Outcome `string`はString、`integer`はSafe Integer、`float`はFinite Number、`boolean`はBoolean、Nullableだけ`null`を許可する
- Outcomeに不足Field、未知Field、Runtime Type不一致があれば`unexpected_response`

### Failure

- Protocol: Status 400、`status === 'error'`、Non-empty String `code`。Raw Bodyや未知DetailはResultへコピーしない
- Rejected: Status 400／401／403／404／409、Operation Responderの`status === 'rejected'`またはOperation成立前Middlewareの既知`status === 'error'` Shapeを検証する。CategoryとHTTP Statusを一致させ、Non-empty String `code`、Optional String `operationId`だけを返す
- Validation: Status 422、`status === 'rejected'`、`category === 'validation'`、`code === 'validation.failed'`、String `operationId`、Array `violations`。各Violationは既知Value Field、Non-empty String `rule`／`code`
- Internal: Status 500、`status === 'error'`、`code === 'internal_error'`、Optional String `operationId`
- Status、Content-Type、JSON、Discriminant、Category、既知Field、Operation固有Fieldの不一致はTransport `unexpected_response`

`operationId`はResponseに存在する場合だけResultへ含める。架空値、空文字、Request情報からの補完を行わない。

既知Response ObjectのUnknown Keyは成功／失敗を問わずDriftとして`unexpected_response`にする。ただしHTTP Headerの追加は無視する。

## Transport and Error Boundary

- Invalid `baseUrl`だけを`invalid_base_url`へ変換する。Value Runtime Type不一致やHeader構築ErrorなどProgrammer Errorを別Transport Codeへ偽装しない
- Injected／Default Fetchがない場合は`missing_fetch`
- SignalがRejected時点で`aborted === true`なら`aborted`、それ以外のThrow／Rejected Promiseは`network_error`
- Response `text()` FailureもRaw Detailを含めず`network_error`とする
- Response Decode Failureは`unexpected_response`
- Transport ResultはCodeだけを持ち、Exception Name／Message／Stack、Raw Response Body、Header、URL、Credential、Request Valueを含めない
- `.fetch()`はHTTP Statusを通常の例外制御へ使わずResult Unionを返す
- Retry、Backoff、Polling、Cacheを行わない

## Determinism and Sensitive Boundary

- Outcome TypeとDecoderはArtifactのField名／Kind／Nullableだけから生成する
- Sensitive InputはWrite-only Value Typeに残るが、Result、Error、Transport、Debug Helperへ混ぜない
- Raw BodyはDecode中だけLocalで扱い、Resultへ含めない
- Contract Hash／Marker、Atomic Output Safety、Non-marker／Symlink拒否をP15-003から維持する
- Generated Bytesへ生成日時、Absolute Path、Environment、Credential、Example、Default実値を含めない

## Acceptance Criteria

- [ ] Operation Objectが`.fetch()`とOperation固有Promise Resultを持つ
- [ ] Browser既定FetchとPer-call Injected Fetchを同じ共通Runtimeで選択する
- [ ] `.toRequest()`／`.url()`／Readonly MetadataのP15-003 Contractが回帰しない
- [ ] Inline Outcome 200、Inline Void 204、Deferred 202をOperation Strategy／Outcome ModeどおりDecodeする
- [ ] Protocol、Rejected、Validation、InternalをStatusとJSON Shapeから厳密Decodeする
- [ ] Missing Fetch、Invalid Base URL、Network、Abort、Unexpected ResponseをTransport Codeで区別する
- [ ] Operation固有Outcome Scalar、Nullable、Validation FieldをRuntime検証する
- [ ] Resultが`ok`／`kind`／`status`でNarrowingできるDiscriminated Unionである
- [ ] Operation IDをResponseに存在する場合だけ保持する
- [ ] Raw Body、Credential、Exception Detail、Sensitive ValueをGenerated Artifact／Resultへ含めない
- [ ] Callable／Thenable、Global Mutable Client、Retry／Polling／Framework依存を追加しない
- [ ] Deterministic Bytes、Marker、Output Safety／Rollbackが回帰しない
- [ ] HTTP Responder、Public PHP API、Migration、Database Schemaを変更しない
- [ ] Required PHP Quality Gateが成功する
- [ ] WorkerはCommitしない
