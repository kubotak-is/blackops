# P16-006: Generated Wait Capability and Frontend CI

Status: Ready

## Goal

全Generated HTTP Operation Objectへ、Terminal Stateまで明示的にStatusを取得する`.wait(operationId, options)`を追加する。

`.wait()`は呼出単位のAbort Signalと有限のDeadlineを必須とし、Serverの`Retry-After`を尊重する。任意Errorを自動Retryせず、Timer／Clock／Fetchを構造型で注入可能にして、DOMなしStrict TypeScript、並行Wait分離、Listener／Timer CleanupをPermanent Frontend FixtureとGitHub Actionsで固定する。

## In Scope

- 全Generated HTTP Operation Objectの`.wait(operationId, options)`
- Required Abort SignalとRequired `maxWaitMilliseconds`
- Immediate Status Query、Terminal停止、Non-terminal Polling
- Server `Retry-After`に従う有限待機
- Abort、Deadline、`poll_timeout`のSafe Result
- Timer／Clock／Fetchの呼出単位構造型注入
- Pre-abort、Sleep中Abort、In-flight Abort、Cleanup
- 401／404／410／5xx／Network／Malformedの即時停止
- 複数`.wait()`の完全分離
- Marker更新、Deterministic Generated Bytes、Drift回帰
- Permanent `tests/Frontend` Strict TypeScript／Node Runtime Evidence
- 既存GitHub Actions Frontend JobへのEvidence統合
- Internal Documentation、Report、TODO、STATE同期

- `.fetch()`からの自動Polling
- `.status()`のResponse Contract、HTTP Resource、Public PHP API変更

## Operation Object Surface

各HTTP Operation Objectへ次を追加する。

```ts
GenerateReport.wait(operationId, {
  signal,
  maxWaitMilliseconds: 30_000,
  fetch,
});
```

- `.wait()`はValueを受けず、Operation固有Status ResultのPromiseを返す
- `operationId`は既存`.status()`と同じCanonical lowercase UUIDv7検証を使う
- `.wait()`はOperation Route Bindingを使わずFramework Status Resourceだけを取得する
- `signal`と`maxWaitMilliseconds`はRequiredで、省略をType Errorにする
- `baseUrl`、Headers、Credentials、Fetchは既存Call Option境界を再利用する
- Clock／TimerはTest可能性のため呼出単位で任意注入できる
- `.status()`は一回取得のまま、`.fetch()`は202後に自動Pollingしない
- Callable、Thenable、Class Constructor、Module Import時の通信を追加しない

## Structural Wait Types

DOM LibraryなしでCompileできる生成型を提供する。概念Shapeは次と同等以上にする。

```ts
type OperationWaitSignal = Readonly<{
  aborted: boolean;
  reason?: unknown;
  addEventListener(type: 'abort', listener: () => void, options?: Readonly<{ once?: boolean }>): void;
  removeEventListener(type: 'abort', listener: () => void): void;
}>;

type OperationWaitClock = Readonly<{
  nowMilliseconds(): number;
}>;

type OperationWaitTimer = Readonly<{
  setTimeout(callback: () => void, milliseconds: number): unknown;
  clearTimeout(handle: unknown): void;
}>;

type OperationWaitOptions = Omit<OperationCallOptions, 'signal'> & Readonly<{
  signal: OperationWaitSignal;
  maxWaitMilliseconds: number;
  clock?: OperationWaitClock;
  timer?: OperationWaitTimer;
}>;
```

実装上の名前は生成可読性のため変更できる。ただしDOMの`AbortSignal`、`EventListener`、`TimerHandler`、NodeJS Namespaceへ型依存せず、既存`.status()`／`.fetch()`へ渡せる最小Abort ShapeとWait購読可能Shapeを型安全に分ける。

- `maxWaitMilliseconds`は正のSafe Integerだけ受理する
- Invalid OptionはFetch／Timerを呼ばずTransport `invalid_wait_options`を返す
- Clockは有限のSafeな非負Millisecondsを返す。Invalid Clock Resultは`invalid_wait_options`とし、Raw値を返さない
- Default Clockは`Date.now()`相当、Default TimerはRuntimeの`setTimeout`／`clearTimeout`相当を使う
- Default Runtime解決に失敗した場合はThrowせずTransport `invalid_wait_options`へ閉じる
- Timer Handle、Deadline、Listener、Polling Stateは各`.wait()` Invocation内だけに保持する

## Wait Result Contract

`.wait()`は既存Operation固有Status Resultを返し、Status Transport Errorへ次を追加する。

- `invalid_wait_options`
- `poll_timeout`

`aborted`は既存Status Transport Error Codeを再利用する。既存`.fetch()`の`OperationTransportError` Unionは変更しない。

- Terminal `completed`／`rejected`／`failed`／`dead_lettered`は取得したResultをそのまま返す
- Authentication／Unavailable／Expired／Internal／Transport Failureは取得したResultをそのまま返す
- Non-terminal `accepted`／`running`／`retry_scheduled`は通常の`.wait()`成功Resultとして返さず、Terminal／Abort／Timeout／Errorまで進む
- `poll_timeout`と`aborted`へOperation ID、Elapsed、Deadline、Signal Reason、Raw Errorを含めない
- Result／Nested Errorは既存方針どおりFreezeする

## Polling Algorithm

1. OptionとOperation IDをLocal検証する
2. Signalが既にAbortedならFetch 0回で`aborted`を返す
3. Clockを一回読み、`start + maxWaitMilliseconds`のDeadlineをInvocation内で固定する
4. `.status()`と同じQueryを直ちに一回実行する
5. Terminalまたは`ok: false`なら直ちに返す
6. Non-terminalならClockを再読し、Remaining Timeを計算する
7. `retryAfterSeconds * 1000`とRemaining Timeの小さい方だけAbort可能に待つ
8. Retry-Afterより先にDeadlineへ到達した場合は追加Fetchせず`poll_timeout`を返す
9. 待機完了後、Signal／Deadlineを再確認して次のStatus Queryへ進む

Deadline判定ではClockの逆行を許容しない。前回値より小さい値または非有限／Unsafeな値は`invalid_wait_options`へ閉じる。Elapsed回数を積算して無限化せず、固定Deadlineを正本にする。

## Abort and Cleanup

- Pre-abortはListener／Timer／Fetchを開始しない
- In-flight Fetchへ同じSignalを渡し、Abort由来のRejected Promiseは`aborted`へ正規化する
- Sleep開始前後にAbort状態を確認し、Lost Wakeupを防ぐ
- Sleep中AbortはTimerをClearし、ListenerをRemoveして`aborted`を返す
- Timer完了時もListenerをRemoveする
- Timer／Listener登録または解除がThrowしてもRaw Detailを露出せず、安全に終了する
- Invocation完了経路のすべてで、そのInvocationが所有するTimer／Listenerを残さない
- 一方のWaitのAbort／Timer／Fetch／Deadlineが別Waitへ影響しない

## Retry and Deadline Rules

- Retry対象はStrict Decode済みのNon-terminal 200だけ
- `Retry-After`の検証は既存`.status()`が所有し、Fallback Intervalを導入しない
- Retry-AfterがRemainingより大きい場合はRemainingだけ待ってTimeoutし、余分なFetchをしない
- DeadlineとAbortが同時に観測された場合はAbortを優先する
- 401、404、410、500、Network、Malformed、Invalid ID、Invalid Optionは一回で停止する
- Retry Count、Backoff、Jitter、Polling CallbackはPublic Optionへ追加しない

## Marker, Determinism, and Drift

Generated Runtime Surface変更としてOwnership Marker Schemaを4へ上げる。Ownership判定だけ既知Legacy 1／2／3／Current 4を受理し、新規／Temporary Tree検証はCurrent 4だけを受理する。

- Frontend Contract Manifest Schemaは上げない
- 同じContractから二回生成したFile Path／Bytesを一致させる
- Current 3 Treeを4へ安全に置換できる
- Unknown Marker、Non-marker、Symlink、Rollback Safetyを維持する
- `frontend:check`は`.wait()`を含むExpected Treeに対するDriftを検出する
- Generated TreeへTimestamp、Absolute Path、Environment、Credential、Fixture Valueを含めない

## Permanent Frontend Evidence

`tests/Frontend`の既存Pinned ToolchainへStatus／Wait Evidenceを恒久追加する。

### Strict TypeScript

- `.status()`の7 State／Error Narrowing
- `.wait()`のTerminal State／Error Narrowing
- Operation固有Typed OutcomeとEmptyOutcome
- `signal`／`maxWaitMilliseconds`省略がCompile Error
- DOM LibraryなしでClock／Timer／Signal構造型を利用できる
- `.fetch()` Result Unionは`poll_timeout`／`invalid_wait_options`を持たない

### Node Runtime

- Immediate Terminalは一回Fetch、Timer 0回
- accepted／running／retry_scheduledを経てTerminalへ到達する
- Retry-After秒をMillisecondsへ変換し、注入Timerへ渡す
- Retry-AfterがRemaining超過ならRemainingだけ待ち、追加FetchなしでTimeout
- Pre-abortはFetch／Timer／Listener 0回
- Sleep中Abort、In-flight Abort、Deadline Timeout
- 401／404／410／500／Network／Malformedは一回で停止
- Invalid ID／Invalid Wait OptionはFetch 0回
- 並行WaitのClock／Timer／Signal／Fetch／Cleanupが分離される
- Success／Abort／Timeout／Error全経路でTimer／ListenerがCleanupされる
- `.fetch()`は自動Status取得しない
- Module Import時にFetch／Timer／Listenerを開始しない
- Result FreezeとSensitive／Raw Error非露出

既存`pnpm --dir tests/Frontend run test`から上記が実行されるようにする。既存`runtime-test.mjs`の拡張で十分なら`.github/workflows/ci.yml`は変更しない。別Scriptを追加した場合もPackage Script経由で既存CI Stepに含まれる構成を優先する。
